#### 📝 提示词结构模板（推荐四段式）
```markdown
# 角色定义

# 示例参考（Few-shot）
```

#### 🎯 分步实操指导（讲师演示+学员跟练）
| 步骤 | 操作指引 | 关键技巧 | 常见错误 |
|------|----------|----------|----------|
| ① 清空默认提示词 | 点击"人设与回复逻辑"→全选删除 | 避免新旧指令冲突 | 直接在原文修改导致逻辑混乱 |
| ② 填写角色定义 | 用"你是..."开头，明确专业边界 | 加入"不是..."排除歧义 | 角色过于宽泛如"全能助手" |
| ③ 设定任务流程 | 用"当...时，你需要..."句式 | 拆解为2-3个可执行步骤 | 任务描述模糊如"帮用户解决问题" |
| ④ 规范输出格式 | 明确"请用...格式回复" | 结合业务场景设计模板 | 格式要求与模型能力不匹配 |
| ⑤ 添加示例参考 | 提供1-2组问答对 | 示例要覆盖高频场景 | 示例与指令矛盾 |

## 🕙 模块三：提示词工程深度实战（9:30-10:30｜60分钟）

### 🔹 9:30-9:50｜提示词设计四大原则（20分钟）

#### 🎯 原则一：角色明确（Role Clarity）
```
❌ 模糊表述："你是一个助手"
✅ 精准定义："你是一位有5年经验的新媒体运营专家，专注小红书爆款内容创作"
💡 技巧：加入"背景+资历+专长"三要素
```

#### 🎯 原则二：任务拆解（Task Decomposition）
```
❌ 笼统指令："帮用户写文案"
✅ 结构化流程：
1. 先询问产品卖点+目标人群+发布平台
2. 再提供3种风格草稿供选择
3. 最后给出优化建议和数据预判
💡 技巧：用"第一步/第二步/最后"明确流程
```

#### 🎯 原则三：格式约束（Format Control）
```markdown
✅ 有效指令：
"请用以下JSON格式回复：
{
  "title": "标题",
  "points": ["要点1", "要点2"],
  "hashtag": ["#标签1", "#标签2"]
}"

❌ 无效指令：
"请规范输出"（模型无法理解"规范"具体指什么）
```

#### 🎯 原则四：边界设定（Boundary Setting）
```
✅ 必加约束：
- "如果用户问题超出[范围]，请礼貌说明并引导至[替代方案]"
- "涉及医疗/法律等专业建议，必须标注'仅供参考，建议咨询专业人士'"
- "不确定答案时，诚实告知而非编造"
```

#### 🚀 技巧1：Few-shot示例增强

```
🎯 实操示例：
```

```
然后再给出最终回复。"
```

### 🔹 10:15-10:30｜参数调优+问题排查（15分钟）

#### ⚙️ 关键参数详解（编辑页→模型设置）
| 参数 | 推荐值 | 调整逻辑 | 业务场景 |
|------|--------|----------|----------|
| Temperature | 0.3-0.7 | 越低越稳定，越高越创意 | 客服Bot用0.3，创意文案用0.7 |
| Top-P | 0.8-0.95 | 控制采样多样性，通常与Temp联动 | 保持回复丰富性同时避免跑偏 |
| 上下文轮数 | 5-10轮 | 越多记忆越好，但增加成本/延迟 | 复杂咨询用10轮，简单问答用5轮 |
| 最大回复长度 | 500-2000字 | 根据业务需求平衡信息量与阅读体验 | 知识科普用长文本，快捷回复用短句 |

#### 🔍 常见问题自查清单
```
❓ Bot回答太冗长？
→ 检查提示词是否添加"简洁回复""分点不超过3条"等约束

❓ Bot频繁答非所问？
→ 1.角色定义是否清晰 2.是否缺少示例参考 3.知识库是否干扰

❓ Bot拒绝回答合理问题？
→ 检查边界约束是否过严，添加"例外情况"说明

❓ 回复格式不稳定？
→ 用"必须用...格式""如果无法...请回复'暂不支持'"强化指令
```

### 🔹 10:45-11:05｜插件核心认知（20分钟）

#### 🌐 插件价值图解
```
🤖 纯大模型Bot：能聊天+有知识 → "大脑"
```
