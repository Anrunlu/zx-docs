### 🔹 14:00-15:30｜实例2：中等复杂度实战

#### 🎯 场景：学生请假审批工作流

```
业务需求：
学生提交请假申请 → 系统校验 → 按天数路由审批 → 通知结果 → 记录归档

📋 工作流架构：
┌─────────────────────────────────────┐
│ Start节点                           │
│ 输入：user_id, leave_type, dates, reason│
└────────┬────────────────────────────┘
         ▼
┌─────────────────────────────────────┐
│ Code节点：日期校验+天数计算         │
│ • 校验：开始日期≥今天，结束≥开始    │
│ • 计算：days = (end-start).days+1  │
│ • 输出：valid, days, error         │
└────────┬────────────────────────────┘
         ▼
┌─────────────────────────────────────┐
│ Condition节点：路由审批级别         │
├─────────────────────────────────────┤
│ 分支A：days≤3 → 辅导员审批         │
│ 分支B：3<days≤7 → 院系审批         │
│ 分支C：days>7 → 院系+补充材料      │
└────────┬────────────────────────────┘
         ▼
┌─────────────────────────────────────┐
│ 分支内子流程（以分支B为例）         │
│ 1. Knowledge节点：查请假政策        │
│ 2. Database节点：插入申请记录       │
│ 3. Plugin节点：发送通知给辅导员    │
│ 4. Variable节点：更新审批链         │
└────────┬────────────────────────────┘
         ▼
┌─────────────────────────────────────┐
│ Merge节点：聚合各分支结果           │
│ • 统一输出格式                      │
│ • 记录审批级别/下一步/政策参考      │
└────────┬────────────────────────────┘
         ▼
┌─────────────────────────────────────┐
│ End节点：返回申请结果               │
│ {                                  │
│   "success": true,                 │
│   "application_id": "APP20260221001",│
│   "approval_level": "院系",         │
│   "next_step": "等待张老师审核",    │
│   "policy": "病假需提供医院证明..." │
│ }                                  │
└─────────────────────────────────────┘
```

#### 🔧 关键配置详解

```
1️⃣ 日期校验代码节点（Python）：
```python
def main(start_date, end_date):
    from datetime import datetime, timedelta
    
    try:
        d1 = datetime.strptime(start_date, "%Y-%m-%d").date()
        d2 = datetime.strptime(end_date, "%Y-%m-%d").date()
        today = datetime.now().date()
        
        # 校验规则
        if d1 < today:
            return {"valid": False, "error": "开始日期不能早于今天"}
        if d2 < d1:
            return {"valid": False, "error": "结束日期不能早于开始日期"}
        if (d2-d1).days > 30:
            return {"valid": False, "error": "单次请假不超过30天"}
        
        days = (d2 - d1).days + 1
        return {"valid": True, "days": days}
    except Exception as e:
        return {"valid": False, "error": f"日期格式错误：{str(e)}"}
```

2️⃣ 条件分支表达式：
```
分支A（辅导员）：{{calc.output.days}} <= 3
分支B（院系）：{{calc.output.days}} > 3 AND {{calc.output.days}} <= 7
分支C（院系+材料）：{{calc.output.days}} > 7
```

3️⃣ 通知插件配置（企业微信）：
```
• 接收人：{{approver_map[{{workflow.approval_level}}]}}
• 消息模板：
  "📋 新请假申请
   申请人：{{start.user_id}}
   类型：{{start.leave_type}}，{{calc.output.days}}天
   原因：{{start.reason}}
   👉 点击审核：[链接]"
• 发送结果：记录到workflow.notification_sent
```

4️⃣ 数据库插入SQL：
```sql
INSERT INTO leave_applications (

```

> 💡 **设计原则**：中等复杂度工作流的核心是"分支管理"。确保每个分支有明确的入口/出口，避免逻辑孤岛；用Variable节点统一管理状态，降低耦合[[3]]。

#### 🔧 调试技巧与最佳实践

```
🛠️ 调试四步法：

1️⃣ 日志追踪（Log节点）
• 在关键节点后添加Log，输出变量值

• 执行时查看"执行日志"面板，定位问题节点

2️⃣ 单步调试（Step-by-Step）
• Coze支持"暂停执行"，逐节点查看输入输出
• 适合：排查数据传递错误/条件判断异常

3️⃣ Mock测试（模拟数据）
• 为插件节点配置"测试响应"，避免依赖外部服务
• 示例：天气插件Mock返回 `{"temp":25, "condition":"晴"}`
• 优势：快速验证逻辑，不受网络/配额限制

4️⃣ A/B测试（策略对比）
• 复制工作流，修改某节点配置（如条件阈值）
• 用相同输入测试，对比输出差异
• 适合：优化路由策略/推荐算法

⚡ 性能优化技巧：

✅ 并行执行：无依赖的节点用并行分支，减少总耗时
✅ 懒加载：非必需数据（如历史工单）按需查询，非启动即查
✅ 结果缓存：Code节点中缓存重复计算结果（如日期解析）
✅ 分页处理：大数据量用分页+Loop，避免单次超时

🔐 安全与合规：

• 敏感数据脱敏：Log节点中不输出手机号/身份证
• 权限校验：Database操作前验证user_id匹配
• 操作审计：关键变更（如删除工单）记录操作人+时间
• 用户授权：涉及外部通知（短信/邮件）需用户确认
```

#### 🔍 常见问题答疑（Q&A精选）

```
Q1：工作流执行超时怎么办？
A：• 检查是否有死循环/无限递归
   • 优化并行分支，减少串行依赖
   • 对耗时插件设置合理timeout+fallback
   • 复杂逻辑拆分为子工作流，按需调用

Q2：如何调试变量传递错误？
A：• 在关键节点后添加Log节点，打印变量值
   • 使用Coze的"执行追踪"功能，可视化数据流
   • 检查变量名拼写（区分大小写）
   • 确认上游节点是否成功执行（status!=error）

Q3：工作流能调用另一个工作流吗？
A：✅ 支持！用"Sub-Workflow"节点
   • 适用场景：复用通用逻辑（如"身份验证"）
   • 配置：指定子工作流ID + 输入参数映射
   • 注意：避免循环调用（A调B，B又调A）

Q4：如何管理工作流版本？
A：• Coze支持工作流"保存为新版本"
   • 建议：大改前复制备份，小调用"另存为v1.1"
   • 发布时标注变更日志（Changelog）
   • 保留1-2个历史版本，便于回滚

Q5：工作流能定时自动执行吗？
A：✅ 支持！用"Trigger"配置
   • 定时触发：cron表达式（如"0 8 * * *"表示每天8点）
   • 事件触发：数据库变更/插件回调/HTTP webhook
   • 适用场景：日报生成/名额释放/数据同步
```

## 📚 附录：工作流开发速查指南

### 🔹 节点选型决策树
```
用户问题/需求
│
├─ 需要调用外部API？ → ✅ Plugin节点
│
├─ 需要查静态知识？ → ✅ Knowledge节点  
│
├─ 需要存/查个人数据？ → ✅ Database节点
│
├─ 需要复杂计算/解析？ → ✅ Code节点
│
├─ 需要条件分支？ → ✅ Condition节点
│
├─ 需要遍历列表？ → ✅ Loop节点
│
├─ 需要暂存中间结果？ → ✅ Variable节点
│
└─ 以上都不需要 → 考虑用提示词+插件，无需工作流
```

### 🔹 变量命名规范
```
✅ 推荐：
• 蛇形命名：user_id, approval_level, total_days
• 语义清晰：workflow.has_error, plugin.weather.temp
• 作用域前缀：start.xxx（输入）, calc.xxx（代码输出）

❌ 避免：
• 无意义缩写：a1, tmp, res
• 中文/特殊字符：用户ID, temp-value
• 重复命名：不同节点用相同变量名导致覆盖
```

### 🔹 错误处理模板
```python
# Code节点通用错误处理框架
def main(input1, input2):
    try:
        # 业务逻辑
        result = do_something(input1, input2)
        return {"success": True, "data": result}
    except ValueError as e:
        # 用户输入错误
        return {"success": False, "error": f"参数错误：{str(e)}", "code": "INVALID_INPUT"}
    except TimeoutError:
        # 外部服务超时
        return {"success": False, "error": "服务繁忙，请稍后重试", "code": "TIMEOUT"}
    except Exception as e:
        # 未知错误
        log(f"Unexpected error: {str(e)}")  # 记录日志
        return {"success": False, "error": "系统异常，已记录", "code": "INTERNAL_ERROR"}
```

### 🔹 性能优化Checklist

□ 分页处理：大数据量用分页+Loop
□ 超时设置：插件节点配置合理timeout
□ 降级策略：关键插件配置fallback方案
